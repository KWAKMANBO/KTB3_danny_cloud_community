# 시딩 삭제(soft-delete) 비율을 25%로 설정한 이유

벤치마크 시딩([`seed.sh`](seed.sh))에서 게시글의 **25%를 `deleted_at`이 채워진 soft-delete 상태**로 생성한다.
```bash
./benchmark/seed.sh 1000000 1 25   # 총 100만 건 중 25만 건(25%) 삭제 처리
```
> 구현: `IF(MOD(n, 100) < 25, ...deleted..., NULL)` — n을 100으로 나눈 나머지가 25 미만인 행을 삭제 처리해 정확히 25%를 재현(랜덤 아님, 결정적).

이 값을 0이 아닌 25%로 정한 데에는 세 가지 이유가 있다.

---

## 1. 인덱스 실험(실험 3)을 유의미하게 만들기 위해 ★ 주된 이유

커서 조회 쿼리는 다음과 같다.
```sql
WHERE deleted_at IS NULL AND post_id < ? ORDER BY post_id DESC LIMIT 20
```

- 삭제가 **0%** 라면 `deleted_at IS NULL`이 모든 행에 참이라 **필터가 무의미**해진다.
  → `(deleted_at, post_id)` 복합 인덱스를 추가해도 PK 단독과 차이가 거의 없어, 실험 3이 "인덱스 효과 없음"으로 김빠진다.
- 삭제 행이 섞여 있어야 "삭제 안 된 행만 골라내는" 비용이 발생하고, 그 비용을 인덱스로 줄일 수 있는지 측정할 수 있다.

즉 **인덱스의 선택도(selectivity)를 실험 가능한 수준으로 만들기 위해** 0이 아닌 비율이 필요했다.

## 2. 현실성 — 너무 극단적이지 않게

- 실제 커뮤니티에는 삭제/숨김 글이 일정 비율 존재한다. **4건 중 1건**은 그럴듯한 수준.
- 90%처럼 과도하게 높으면 "결과를 유리하게 조작했다"는 인위적 인상을 준다.
- 25%는 **현실적이면서 효과도 관측 가능한 중간 지점**이다.

## 3. 실험 1에서 OFFSET의 낭비를 드러내는 효과

실험 1 결과에서 `depth=100,000`일 때 OFFSET의 스캔 행수는 **133,346행**이었다. 이는 25% 삭제에서 직접 나온 수치다.
```
100,020 (건너뛸 노출 글) ÷ 0.75 (노출 비율) ≈ 133,360
```
삭제된 25%는 OFFSET 카운트에는 잡히지 않지만 **읽기는 발생**하므로, 건너뛸 양보다 약 33% 더 스캔한다.
→ **"삭제 행이 OFFSET 낭비를 증폭한다"** 는 설명 포인트가 이 비율에서 나온다.

또한 `depth=900,000`에서 OFFSET이 `rows=0`(결과 없음)을 반환한 것도 이 비율 때문이다.
노출 글이 750,000건뿐이라 offset 900,000이 데이터 끝을 넘어서지만, 그 사실을 확인하려 100만 행 전체를 스캔한다.

---

## 한계와 조정 방법

- 25%가 절대적 정답은 아니며, 실험 설계상 선택한 값이다.
- **너무 낮으면**(예: 5%) 실험 3의 인덱스 효과가 약하게 나올 수 있다.
- **더 극적인 인덱스 효과**를 원하면 비율을 높여 재시딩할 수 있다.
  ```bash
  ./benchmark/seed.sh 1000000 1 50   # 삭제 50%
  ```

## 요약

| 이유 | 내용 |
|------|------|
| 인덱스 실험 | `deleted_at IS NULL` 필터에 선택도를 부여해 `(deleted_at, post_id)` 인덱스 효과 측정 가능 |
| 현실성 | 실서비스의 삭제 글 비율을 가정한 그럴듯한 값(4건 중 1건) |
| OFFSET 낭비 | 삭제 행이 offset 스캔량을 증폭(100k→133k)하고, 끝을 넘는 페이지 풀스캔을 유발 |

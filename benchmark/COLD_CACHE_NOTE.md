# depth=0 커서가 유독 느린 이유 — 콜드 캐시(warm-up)

실험 1([`RESULTS.md`](RESULTS.md))에서 `depth=0`의 커서 값만 다른 depth보다 크게 나온다.

- `depth=0` OFFSET = **0.077 ms**
- `depth=0` CURSOR = **3.83 ms** ← 유독 느림
- (참고: `depth=10000`은 OFFSET 3.99ms / CURSOR 0.18ms 로, 거기선 커서가 훨씬 빠름)

이는 **커서가 느려서가 아니라, 이 값이 스크립트 전체에서 "첫 번째 커서 쿼리"라서** 발생하는 콜드 캐시(warm-up) 현상이다.

---

## 원인: InnoDB 버퍼 풀(캐시)

MySQL(InnoDB)은 데이터/인덱스 페이지를 **버퍼 풀(메모리)** 에 캐싱한다.
- 필요한 인덱스 페이지가 **메모리에 있으면** → 빠름 (~0.05 ms)
- **메모리에 없으면** → 디스크에서 처음 읽어옴 → 그 I/O가 몇 ms

`depth=0`의 커서 쿼리는 이 실행에서 **최초로 실행된 커서 쿼리**라, 필요한 인덱스 페이지(특히 범위 스캔의 시작점을 찾는 B-tree 하강 경로)가 아직 버퍼 풀에 없어 디스크에서 읽느라 3.83ms가 걸렸다. 실제 커서 성능이 아니라 **최초 1회 워밍업 비용**이다.

## 증거 1 — 바로 다음 쿼리부터 급락

| depth | CURSOR actual time | 비고 |
|------:|---:|---|
| 0 | 3.83 ms | 콜드 (첫 커서 쿼리) |
| 1,000 | 0.20 ms | 캐시됨 |
| 10,000 | 0.18 ms | |
| 100,000 | 0.094 ms | |
| 500,000 | 0.052 ms | |
| 900,000 | 0.057 ms | |

같은 형태의 커서 쿼리인데 두 번째부터 0.2ms 이하로 뚝 떨어진다. 커서 방식 자체가 느린 것이라면 모든 depth에서 느려야 한다.

## 증거 2 — 두 번의 실행에서 "같은 자리"만 느림

- 1차 실행: `depth=0` 커서 = **1.12 ms**
- 2차 실행: `depth=0` 커서 = **3.83 ms**

두 번 모두 **"첫 커서 쿼리"에서만** 값이 튀었다. 커서 방식 자체의 문제라면 매 depth마다 느려야 하는데, 항상 **첫 쿼리**에서만 발생한다 → 워밍업 효과라는 결정적 증거.

## OFFSET(depth=0)은 왜 안 느렸나

스크립트는 depth마다 **OFFSET → CURSOR 순**으로 실행한다. OFFSET 쿼리가 먼저 돌면서 맨 앞 21행만 읽는 가벼운 작업으로 일부 페이지가 캐시에 올라왔고, 커서의 범위 스캔은 시작 지점을 찾는 tree descent 페이지들을 추가로 건드리는데 그 경로가 이 실행에선 콜드였다.

---

## 결론

- `3.83 ms`는 **커서가 느린 것이 아니라, 첫 쿼리라 디스크에서 읽은 워밍업 비용**이다.
- 두 번의 실행 모두 "첫 커서 쿼리"에서만 튀었다는 점이 이를 뒷받침한다.
- 실제 커서 성능 = 그 뒤의 **0.05~0.2 ms 상수**로 봐야 한다.

## 없애는 방법 (측정 정밀도 개선)

측정 반복문 시작 전에 **워밍업 쿼리 한 번**을 던지고 그 결과를 버리면, `depth=0` 커서도 캐시된 상태에서 측정되어 ~0.05ms대로 나온다. (`compare_pagination.sh`의 depth 루프 앞에 커서 쿼리 1회 추가)

> 벤치마크 일반 원칙: **콜드/웜 상태를 구분하고, 워밍업 후 안정된 값을 측정한다.** 첫 실행값은 버린다.
